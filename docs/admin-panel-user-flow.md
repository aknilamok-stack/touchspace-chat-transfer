# TouchSpace Admin Panel User Flow

## Сценарий 1. Подтверждение регистрации

1. Админ открывает раздел `/admin/registrations`.
2. На экране видит список заявок в статусе `pending`.
3. Открывает нужную заявку.
4. Проверяет имя, email, компанию и роль.
5. Выбирает одно из действий: `approve` или `reject`.
6. При необходимости добавляет комментарий или причину отказа.
7. Система меняет статус заявки на `approved` или `rejected`.
8. Если заявка подтверждена, профиль пользователя получает активный доступ.
9. После `approved` пользователь может войти в систему на следующем этапе интеграции auth.

## Сценарий 2. Ручное создание пользователя

1. Админ открывает раздел `/admin/users`.
2. Нажимает действие “создать пользователя”.
3. Заполняет имя, email, компанию и роль.
4. Назначает нужный статус доступа.
5. Система создаёт профиль вручную через admin API.
6. Новый пользователь появляется в общем списке пользователей.

## Сценарий 3. Просмотр проблемных диалогов

1. Админ открывает раздел `/admin/sla` или `/admin/dialogs`.
2. Фильтрует проблемные кейсы по признаку `SLA breached`.
3. Открывает конкретный диалог.
4. Смотрит историю сообщений, системные события и supplier request.
5. Сравнивает время первого ответа менеджера и время ответа поставщика.
6. Понимает, кто именно задержал коммуникацию.
7. Использует вывод для операционного разбора.

## Сценарий 4. Анализ эффективности менеджера

1. Админ открывает `/admin/analytics/managers`.
2. Выбирает период: день, неделя, месяц или custom.
3. Находит нужного менеджера в сводной таблице.
4. Открывает его карточку или detail endpoint.
5. Смотрит:

- количество диалогов в работе
- количество обработанных диалогов
- среднее время первого ответа
- среднее время закрытия
- количество SLA просрочек
- количество эскалаций к поставщику

6. При необходимости переходит к связанным диалогам.
7. Делает вывод по загрузке и качеству работы менеджера.

## Сценарий 5. Анализ эффективности поставщика

1. Админ открывает `/admin/analytics/suppliers`.
2. Выбирает период.
3. Находит нужного поставщика.
4. Открывает карточку поставщика или detail endpoint.
5. Смотрит:

- количество полученных запросов
- количество отвеченных запросов
- среднее время ответа
- количество SLA просрочек
- количество связанных диалогов
- top причины обращений

6. При необходимости переходит в связанные запросы и диалоги.
7. Использует данные для оценки качества работы поставщика.

## Общий admin flow по системе

1. Админ начинает день с `/admin`, чтобы увидеть общую картину.
2. Проверяет новые регистрации и pending approvals.
3. Переходит в `/admin/users`, если нужно выдать доступ вручную или изменить статус.
4. Контролирует проблемные кейсы через `/admin/dialogs` и `/admin/sla`.
5. Использует `/admin/analytics`, `/admin/analytics/managers` и `/admin/analytics/suppliers` для управленческих выводов.
6. На следующем этапе сможет использовать те же экраны как основу для AI-insights и quality review.
