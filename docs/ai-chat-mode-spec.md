# AI Chat Mode MVP

## Как работает AI-режим

В TouchSpace остаётся один общий диалог на базе `Ticket`.

Внутри этого же диалога теперь могут находиться:
- сообщения клиента
- сообщения менеджера
- сообщения поставщика
- сообщения AI-помощника
- системные события

Клиент может включать и выключать `AI-помощник` прямо в текущем чате.

Когда AI включён:
- в историю добавляется системное сообщение `AI-помощник подключён к диалогу`
- новые сообщения клиента сохраняются как обычно
- backend вызывает OpenAI
- ответ AI сохраняется как отдельный `Message` с `senderType=ai`
- диалог остаётся доступным менеджеру для просмотра
- диалог не должен шуметь в основном входящем пуле менеджера как обычный новый чат

Когда AI выключен:
- в историю добавляется системное сообщение `AI-помощник отключён. Диалог снова ведёт менеджер`
- `Ticket` возвращается в manager-mode
- новые сообщения снова идут в обычный поток менеджера

Если AI понимает, что нужен человек:
- AI отвечает клиенту
- в историю добавляется системное событие передачи
- диалог переводится обратно на менеджера
- такой диалог снова становится обычным входящим

## Какие поля добавлены

### Ticket
- `conversationMode`
- `currentHandlerType`
- `aiEnabled`
- `aiActivatedAt`
- `aiDeactivatedAt`
- `handedToManagerAt`
- `aiResolved`

### Message
Используется существующая схема:
- `senderType`: теперь поддерживает `ai` и `system`
- `senderRole`: теперь может быть `ai`
- `messageType`: используется `text`, `system`, `ai_response`

## Какие endpoint’ы добавлены

### Ticket mode
- `POST /tickets/:id/ai/enable`
- `POST /tickets/:id/ai/disable`

### Existing message flow
- `POST /messages`
  - если `ticket.aiEnabled=true` и пишет клиент, backend сохраняет сообщение клиента, затем генерирует и сохраняет AI-ответ в тот же диалог

### Ticket create flow
- `POST /tickets/with-first-message`
  - поддерживает `aiEnabled`
  - позволяет стартовать новый клиентский диалог сразу в AI-режиме

## Как переключается диалог

### Клиент включает AI
1. frontend вызывает `POST /tickets/:id/ai/enable`
2. backend обновляет `Ticket`
3. backend пишет системное событие
4. новые сообщения клиента начинают идти в AI-слой

### Клиент выключает AI
1. frontend вызывает `POST /tickets/:id/ai/disable`
2. backend обновляет `Ticket`
3. backend пишет системное событие
4. диалог снова становится manager-mode

### AI передаёт менеджеру
1. клиент пишет в AI-режиме
2. backend сохраняет сообщение клиента
3. AI формирует ответ и флаг `shouldHandoff`
4. если нужен человек, backend:
   - отключает AI-mode
   - ставит `currentHandlerType=manager`
   - пишет системное событие передачи
   - возвращает диалог в обычный входящий поток

## Что осталось на следующий этап

- более умная автоэскалация по нескольким правилам и confidence
- расширение AI-аналитики в админке до уровня отдельных cohort/retention-срезов
- настройка системного prompt под каталог, сроки, наличие и правила TouchSpace
- сохранение structured AI outcome для BI и admin analytics
- ручное включение AI менеджером со своей стороны

## Что уже выведено в админку

В разделе `Админка -> Инсайты` доступны базовые AI-метрики:
- сколько диалогов шло через AI
- доля AI-чатов за период
- сколько диалогов AI закрыл без менеджера
- сколько диалогов AI передал менеджеру
- по каким темам и триггерам клиенты чаще используют AI
